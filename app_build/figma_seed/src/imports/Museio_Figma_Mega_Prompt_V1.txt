GLOBAL LOGIC RULES TO REPRESENT IN THE PROTOTYPE
1. Session gating
- Unauthenticated web users land on a marketing/landing page or auth
- Authenticated users go to /app/home
- Protected routes require a valid session
- Sessions near expiry are treated as invalid for route access

2. Onboarding gating
- There is a welcome/setup popup for incomplete setup
- The popup visually emphasizes three required areas: Personal account, Invoicing Details, Bank Account
- As-Is logic inconsistency: the popup gating effectively checks profile plus bank completion, but invoice sending also requires a saved invoice signature
- Represent both truths in the prototype: onboarding popup and stricter invoice-send validation
- Include Skip for now and Don’t show this again options

3. Public portfolio publishing rules
- Portfolio edit mode requires a username/public handle
- If username is missing, show Username Required dialog before portfolio can be published or shared
- Live portfolio hides empty optional sections
- Only show tab navigation in live portfolio if enabled sections actually contain content

4. Availability and booking rules
- Weekly availability editor exists
- Booking availability is constrained by working hours, vacations, existing jobs, and pending booking requests
- Default availability settings to represent in UI copy and settings:
  - minimum notice: 24 hours
  - booking buffer: 15 minutes
  - breaks enabled: off by default
  - default break duration: 60 minutes
- Public booking is multi-step and date/time driven

5. Quote and booking-request rules
- Booking requests arrive publicly and appear in artist Requests inbox
- Artist can quote, decline, or remove request
- Client may accept or decline quote via email-link flow
- Signed quote response tokens are valid for 30 days
- Accepted quote creates an upcoming itemized job and seeds a default line item named “DJ Service”
- As-Is logic quirk: quote acceptance creates the job, but source request does not appear fully normalized to an “accepted” status; model this as an annotation in the design spec notes

6. Jobs rules
- Jobs statuses seen in product/domain: requested, drafted, upcoming, past, invoice-sent, paid, deleted
- Visible Jobs tabs in current UI: Requests, Upcoming, Invoiced, Paid, Drafts
- There is also a hidden past-content state in code; include it as a state or admin/internal variant, but do not place it as a visible primary tab unless clearly annotated
- Multi-day jobs appear on every date between start and end date inclusive
- New jobs and edits support line items, tax toggles, discount percent, notes, and client selection
- Max line items in current UI: 10

7. Invoicing and payment rules
- Invoices can only be sent when profile, bank details, and invoice signature are present
- Default invoice due term is 14 days
- Payment path supports Stripe payment collection and bank details display
- Overdue reminders are rate-limited to once every 7 days
- Paid state is ultimately reconciled from payment/webhook logic

8. Notifications rules
- Email and push notifications exist
- Booking request insert can trigger notification behavior
- Invoice send and overdue reminder behavior should be represented as side-effect notes and success states

AS-IS DESIGN CHARACTERISTICS TO PRESERVE IN THE FIRST PASS

- Large empty-state illustrations and messages
- Purple active states across tabs, headers, chips, and CTAs
